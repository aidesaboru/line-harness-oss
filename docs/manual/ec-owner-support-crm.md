# ECオーナー通達LINE サポートCRM運用マニュアル

## 目的

このマニュアルは、ECオーナー通達LINEの問い合わせ対応を、LINE公式アカウント、スプレッドシート、Slackに分散させず、LINE Harness上で一元管理するための運用手順です。

検証環境で問題なく動くことを確認してから、本番LINE公式アカウントへ段階的に切り替えます。

## ログイン（初回セットアップ）

1. 管理者からAPIキーを受け取る（スタッフ追加時に一度だけ発行されます）
2. 管理者から共有された管理画面URLの `/login` を開く
3. APIキーを入力してログインする

APIキーはパスワードと同じ扱いです。他人へ転送せず、漏れた可能性がある場合は管理者にキーの再生成を依頼してください。

ログインすると、サポートCRMの「自分宛」が自分のスタッフ名を基準に表示されます。

## 最初に見る画面

- チャット対応: 管理画面URLの `/chats`
- 案件管理: 管理画面URLの `/support`

通常対応は、まずチャット対応画面で未対応と24時間超過を確認します。

案件として管理が必要な問い合わせは、サポートCRM画面で案件化します。

## 日次の確認順

1. チャット対応画面を開く
2. 並び順を「24時間超過」または「未対応優先」にする
3. 24h超過バッジが付いているトークを先に確認する
4. 未読・対応中のトークへ返信する
5. 返信だけで終わらないものはサポートCRMで案件化する
6. サポートCRMで「期限超過」「担当者なし」「エスカレ」を確認する
7. 完了したものはステータスを「完了」にする

## チャット対応

チャット一覧では、解決済み以外で最終メッセージから24時間以上経っているトークに、24h超過の目印が表示されます。

長いトークでは、メッセージ欄上部の「過去のメッセージを読み込む」から古い履歴を追加で確認できます。

24h超過が出ている場合は、次の順で対応します。

1. トーク内容を確認する
2. 返信できる内容ならすぐ返信する
3. 確認が必要ならステータスを「対応中」にする
4. 調査や二次確認が必要ならサポートCRMで案件化する
5. 返信後に完了できる場合は「解決済」にする

並び順は以下を使い分けます。

| 並び順 | 使う場面 |
|---|---|
| 新しい順 | 通常の直近確認 |
| 古い順 | 古い未処理を上から潰す |
| 24時間超過 | 24時間以上止まっているものを先に確認 |
| 未対応優先 | 未読、24時間超過、対応中をまとめて確認 |

## 案件化する基準

次のいずれかに当てはまる場合は、チャットだけで終わらせず案件化します。

- 一次対応だけでは判断できない
- 返金、報酬、契約、権利侵害、クレームなど慎重な確認が必要
- Slackや二次対応者へ確認が必要
- 24時間以上止まりそう
- 顧客情報、対応期限、対応履歴を残す必要がある

## サポートCRMの使い方

サポートCRMでは、上部の数字カードから優先キューを確認します。

カードを押すとその条件で絞り込まれ、もう一度押すと解除されます。

| キュー | 意味 |
|---|---|
| 未完了 | 完了していない案件 |
| 期限超過 | 設定期限を過ぎた案件 |
| 24h滞留 | 更新から24時間以上動きがない案件 |
| 担当者なし | 一次担当が決まっていない案件 |
| エスカレ | 二次対応へ確認中の案件 |
| 自分宛 | ログイン中スタッフ名が二次対応先に入っている案件 |
| 顧客返信待ち | 顧客からの返答待ち案件 |
| 完了 | 完了済みの案件（参照用） |

案件を開いたら、画面上部の「まず確認する4点」欄を見ます。

1. ステータス
2. 優先度
3. 一次担当
4. 期限

この4点は案件詳細の最上部にまとまっており、その場で変更できます。ただしstaff権限では、優先度、一次担当、期限は読み取り専用です。担当割りや期限の変更が必要な場合は owner/admin に依頼します。

期限欄の下の「今日18時」「明日10時」「+24h」を押すと、よく使う期限を1クリックで入れられます。

問い合わせ内容は「問い合わせ要約」に短く残します。

社内だけで見る内容は「内部メモ」に残します。

顧客へ送る文章の下書きは「顧客向け返信案」に残します。返信するときは「チャットで返信」を押すと、返信案がチャット入力欄に入った状態でトークを開けます。

チャット画面で送信すると、案件ステータスは「顧客返信待ち」になります。テキストだけでなく画像だけで返信した場合も同じ扱いです。LINEの送信履歴だけでなく案件履歴にも「チャットで顧客返信を送信しました」と残ります。「返信案をコピー」は、別の場所へ貼り付けたい場合の予備操作です。

完了済み案件では「チャットで返信」は使いません。追加返信が必要な場合は、先に案件を再オープンしてから返信します。

### 新規案件の作り方

1. 画面右上の「新規案件」を押す
2. LINE会話を選ぶ（選ぶと最終メッセージが要約に自動で入る）
3. 件名・種別・優先度・一次担当・期限を入れて「案件化する」を押す

件名が空欄でも、要約から自動で件名が付きます。

新しいLINE会話を誰へ渡すか決める案件化は owner/admin が行います。staff権限では「新規案件」は表示されず、自分に関係する既存案件の対応、エスカレ回答、マニュアル検索・紐付けを行います。

### 編集の保存

staff権限で編集できるのは、ステータス、次回確認、問い合わせ要約、内部メモ、顧客向け返信案、対応結果メモ、マニュアル紐付けです。

件名、種別、優先度、一次担当、二次対応先、期限、顧客番号、法人名、担当者名、店舗名、契約種別は owner/admin が管理します。

入力を変更すると、画面下に「未保存の変更があります」のバーが出ます。

「保存」を押すか、⌘S（WindowsはCtrl+S）で保存します。

未保存のまま別の案件へ移動しようとすると確認が出るため、編集が黙って消えることはありません。

### よく使うボタン

| ボタン | 動き |
|---|---|
| 対応開始 | 未対応・再オープンの案件を対応中にする |
| 完了にする… | 完了条件の確認と対応結果メモの入力後に完了へ |
| 再オープン | 完了後に再連絡が来た案件を再オープンにする |
| チャットを開く | この案件のLINEトークを開く |
| チャットで返信 | 顧客向け返信案をチャット入力欄へ引き継ぐ |
| 返信案をコピー | 顧客向け返信案をコピーする |

## エスカレーション

一次対応で判断できない場合は、右側のエスカレーション欄を使います。

1. 二次対応先を入力する（案件の「二次対応先」が入っていれば自動で引き継がれます）
2. L2またはL3を選ぶ
3. 期限を入れる（プリセットで1クリック入力可）
4. 確認してほしい要点を書く
5. 「エスカレ作成」を押す

二次対応先と要点が両方入るまで「エスカレ作成」は押せません。

staff権限では、案件に設定済みの二次対応先へ「確認してほしい要点」だけを送れます。二次対応先、L2/L3、回答期限をstaffが新しく指定することはできません。設定が必要な場合は owner/admin に依頼します。

エスカレ一覧は回答待ちが上に並び、期限を過ぎたものは赤く表示されます。

二次対応者から回答が来たら、回答要点を入力して「回答済み」にします。

回答だけでは足りない場合は「差し戻し」にします。

staff権限では、既存エスカレーションの回答要点入力、回答済み、差し戻しができます。二次対応先、レベル、回答期限、確認してほしい要点の変更は owner/admin が行います。

## 自分宛エスカレの確認

二次対応者は、サポートCRM上部の「自分宛」を開きます。

「自分宛」には、ログイン中スタッフの名前が二次対応先に入っている未完了案件が表示されます。

自分宛案件を開いたら、次の順で確認します。

1. 期限を見る
2. 確認してほしい要点を見る
3. 会話・履歴タブで直近のやり取りを見る
4. 回答要点を書く
5. 判断できたら「回答済み」にする
6. 情報が不足している場合は「差し戻し」にする

顧客へ直接返信する前提ではなく、原則として二次対応者は回答要点を残し、最終返信はフロント担当が確認して送ります。

## エスカレ共有文

エスカレーション欄の「エスカレ共有文」から、案件情報、問い合わせ要約、確認してほしいこと、直近会話抜粋をまとめてコピーできます。

Slackや社内確認へ一時的に共有する場合は、この共有文を使います。

共有文を使うことで、担当者ごとの説明の抜け漏れや、伝言ゲームによる認識ズレを減らします。

## マニュアル検索

右側のマニュアル欄で、カテゴリやキーワードから対応マニュアルを探します。検索は入力すると自動で絞り込まれます。

一次対応者・二次対応者は、マニュアルの検索と案件への紐付けができます。マニュアル本文の追加、更新、無効化は owner または admin が右側のマニュアル欄から行います。

マニュアルを新規作成・更新するときは、タイトルと本文が必須です。リンクは任意ですが、入力する場合は `http://` または `https://` から始まるURLだけ保存できます。本文には、あとから別の担当者が同じ判断を再現できるように、対応手順か判断基準を残します。

該当するマニュアルがあれば、「紐付け」で案件に紐付けます。

紐付け済みのマニュアルは欄の上部にまとまり、「解除」で外せます。

紐付けたマニュアルは、後から同じ案件を見た人が判断根拠を追えるようにするためのものです。

## 完了条件

案件は、次の状態になったら「完了」にします。

- 顧客への返信が完了している
- 二次対応の確認が完了している
- 社内メモに判断理由が残っている
- 必要なマニュアルが紐付いている
- 再確認予定が不要になっている

「完了にする…」ボタンを押すと、この完了条件と「対応結果メモ」の入力欄が表示されます。対応結果メモを入れないと完了にできません。

保留にする場合も、保留理由の内部メモと次回確認日時が必要です。

完了後に再度連絡が来た場合は、直接「未対応」に戻さず「再オープン」ボタンで扱います。未完了の案件は再オープンにできません。

完了済み案件で二次回答や差し戻しを追加する場合も、先に案件を再オープンしてからエスカレーションを更新します。

## 操作のコツ

- 検索は入力すると自動で絞り込まれます（ボタン不要）
- 案件一覧は ↑↓ キーで移動できます
- ⌘S（Ctrl+S）でいつでも保存できます
- 一覧と件数は60秒ごとに自動更新されます（編集中は止まります）
- 一覧の並び替えは「更新が新しい順」「期限が近い順」「優先度順」から選べます

## 本番切り替え前の確認

本番LINE公式アカウントへ切り替える前に、最低限次を確認します。

実際の切替判断では、[本番投入前チェックリスト](./ec-owner-support-crm-release-checklist.md) を使って、自動検査、画面確認、rollback手順までまとめて確認します。

- 検証用LINE公式アカウントで受信できる
- チャット画面で返信できる
- サポート案件からチャットへ移動し、テキスト返信と画像返信のどちらでも案件履歴が残る
- 24時間超過がチャット一覧で目立つ
- チャット一覧の並び替えができる
- サポートCRMで案件化できる
- 担当者、期限、ステータスを保存できる
- エスカレーションを作成できる
- 「自分宛」でログイン中スタッフのエスカレ案件を確認できる
- staff権限でログインした場合、自分に関係する案件と紐づくチャットだけが表示される
- エスカレ共有文をコピーして、必要情報がまとまっていることを確認できる
- マニュアル検索と紐付けができる
- owner/adminでマニュアル作成・編集時に、タイトル、本文、URL形式の保存前チェックが効く
- 完了、再オープンの運用が確認できる

画面で確認する前に、プリフライトでもAPIと権限を確認します。

### 普段の確認

日次確認やリリース前の軽い確認では、次を実行します。

```bash
SUPPORT_CRM_API_URL=https://your-worker.example.com \
SUPPORT_CRM_ADMIN_ORIGIN=https://your-admin.example.com \
SUPPORT_CRM_LINE_ACCOUNT_ID=本番LINE公式アカウントID \
SUPPORT_CRM_OWNER_API_KEY=ownerのAPIキー \
SUPPORT_CRM_STAFF_API_KEY=staffのAPIキー \
corepack pnpm preflight:support-crm
```

このチェックでは、管理画面ログイン用CORS、ログイン中の権限、サポートCRMの要約・案件一覧・マニュアル検索・チャット一覧、staffによる案件作成禁止・担当変更禁止・エスカレ担当指定禁止・マニュアルの作成/更新/無効化禁止をまとめて確認します。staffの禁止チェックは、ガードが壊れていても実データを作成・更新しにくい不正入力で実行します。`SUPPORT_CRM_STAFF_VISIBLE_FRIEND_ID` がある場合は、未対応のチャット `messageType` が送信前検証で止まることも確認します。`SUPPORT_CRM_ADMIN_ORIGIN` は任意ですが、設定するとブラウザログインに必要な `Access-Control-Allow-Credentials` も確認できます。

失敗した場合は、出力末尾の `Failures to fix` を見ます。そこに、失敗したチェック名と次に確認する場所が表示されます。`Skipped optional checks` は、追加のfixture IDを設定すると確認できる任意チェックです。

### 本番切替前の最終確認

本番切替前は、任意チェックのスキップも失敗扱いにします。staffの表示範囲、未完了案件の再オープン禁止、完了済み案件からの返信禁止まで確認するため、次を実行します。

fixture IDの候補は、D1から読み取り専用で拾えます。`SUPPORT_CRM_STAFF_NAME` は、staff APIキーで `/api/staff/me` に出る名前と同じものを指定します。

```bash
SUPPORT_CRM_LINE_ACCOUNT_ID=本番LINE公式アカウントID \
SUPPORT_CRM_STAFF_NAME=staffのスタッフ名 \
SUPPORT_CRM_D1_ENV=production \
corepack pnpm support-crm:fixtures
```

Cloudflareへ接続せず、実行されるSQLだけ確認したい場合は `corepack pnpm support-crm:fixtures:sql` を使います。

既存データだけではstrict Preflight用のfixtureが足りない場合は、検証用D1にsynthetic fixtureを作ります。先に `corepack pnpm support-crm:fixtures:seed-sql` でSQLを確認し、問題なければ `SUPPORT_CRM_FIXTURE_WRITE=1 corepack pnpm support-crm:fixtures:seed` を実行します。このseedはstaff、友だち、サポート案件、チャット一覧用メッセージだけを作り、LINE送信はしません。検証後は `SUPPORT_CRM_FIXTURE_WRITE=1 corepack pnpm support-crm:fixtures:cleanup` で同じprefixのfixtureを削除し、`corepack pnpm support-crm:fixtures:verify-cleanup` でstaff、案件、イベント、メッセージ、友だち、チャットの残り件数がすべて0であることを確認します。SQLだけ見たい場合は `corepack pnpm support-crm:fixtures:verify-cleanup-sql` を使います。

```bash
SUPPORT_CRM_API_URL=https://your-worker.example.com \
SUPPORT_CRM_ADMIN_ORIGIN=https://your-admin.example.com \
SUPPORT_CRM_LINE_ACCOUNT_ID=本番LINE公式アカウントID \
SUPPORT_CRM_OWNER_API_KEY=ownerのAPIキー \
SUPPORT_CRM_STAFF_API_KEY=staffのAPIキー \
SUPPORT_CRM_REQUIRE_FULL_COVERAGE=1 \
SUPPORT_CRM_STAFF_VISIBLE_CASE_ID=staffが見えてよい案件ID \
SUPPORT_CRM_STAFF_FORBIDDEN_CASE_ID=staffが見えてはいけない案件ID \
SUPPORT_CRM_STAFF_NON_RESOLVED_CASE_ID=staffが見えてよい未完了案件ID \
SUPPORT_CRM_STAFF_RESOLVED_CASE_ID=staffが見えてよい完了済み案件ID \
SUPPORT_CRM_STAFF_VISIBLE_FRIEND_ID=staffが見えてよい友だちID \
SUPPORT_CRM_STAFF_FORBIDDEN_FRIEND_ID=staffが見えてはいけない友だちID \
SUPPORT_CRM_STAFF_RESOLVED_FRIEND_ID=完了済み案件に紐づく友だちID \
corepack pnpm preflight:support-crm
```

`SUPPORT_CRM_REQUIRE_FULL_COVERAGE=1` を付けると、`Skipped optional checks` が1つでも残っている場合に `preflight: full coverage required` が失敗として出ます。つまり「まだ確認していない範囲がある」ことに気づけます。

`SUPPORT_CRM_STAFF_NON_RESOLVED_CASE_ID` を設定すると、未完了案件を誤って再オープンにできないことも確認します。

`SUPPORT_CRM_STAFF_RESOLVED_CASE_ID` と `SUPPORT_CRM_STAFF_RESOLVED_FRIEND_ID` を両方設定すると、完了済み案件から顧客返信を送ろうとしても、LINE送信前の検証で止まることを確認します。この確認は送信前検証だけを使うため、実際のLINEメッセージは送られません。

`SUPPORT_CRM_STAFF_VISIBLE_FRIEND_ID` を設定すると、サポート担当が見えるチャットであっても、未対応の `messageType` は送信前検証で止まることを確認します。この確認も送信前検証だけを使うため、実際のLINEメッセージは送られず、チャット行も作りません。

この確認が終わるまでは、既存の本番LINE公式アカウントの運用は切り替えません。

## スタッフ追加

スタッフは、オーナー権限のアカウントで管理画面のスタッフ管理から追加します。

追加時は、スタッフ名、メール、権限を設定します。staff権限のスタッフ名は空欄にしません。サポートCRMの「自分宛」や表示範囲は、このスタッフ名と案件の担当者名・エスカレ先名を照合して判定します。

| 権限 | 使う人 |
|---|---|
| owner | 全体管理、スタッフ追加、重要設定を行う人 |
| admin | 運用管理者、全体状況を確認する人 |
| staff | 一次対応者、二次対応者、個別案件を見る人 |

スタッフ作成後に表示されるAPIキーを本人へ共有します。

本人はログイン画面でAPIキーを入力します。

ログイン後、「自分宛」はそのスタッフ名を基準に表示されます。Preflightでは、staff APIキーのログイン情報にスタッフ名が入っていることも確認します。

staff権限では、自分が作成・担当・エスカレ先になっている案件と、その案件に紐づくチャットだけが表示されます。

owner/adminは全案件と全チャットを確認できます。新規問い合わせを誰へ渡すか判断する担当者は、ownerまたはadminで確認してください。
